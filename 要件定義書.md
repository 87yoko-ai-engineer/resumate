# AI履歴書作成サービス — 要件定義書 & MVP実装計画

> 本ドキュメントは壁打ち（ブレインストーミング）でまとめた内容を整理した、
> 要件定義と実装計画です。

---

## 1. 背景・目的（Context）

文章力に自信がない転職者でも「採用者に響く履歴書」が作れるサービスを作りたい。
左にAIチャットボット、右に履歴書プレビューを配置し、AI（Claude API）が対話で
情報を聞き出しながら、志望動機・職歴・自己PRを **ブラッシュアップして反映** する点が核。

本プロジェクトは **学習・ポートフォリオ目的**。最重要なのは「途中で止まらず確実に
完成させる」こと。技術的な見せ場（Claude APIのtool use、リアルタイムプレビュー、
PDF生成）を作り込みつつ、スコープを絞り切る。

---

## 2. 設計方針（重要な前提）

### 2.1 ログイン機能・データベースは使わない（ステートレス構成）

- **ユーザー認証（ログイン）を実装しない。**
- **サーバー側のデータベースを持たない。** サーバーは状態を保持しないステートレス構成。
- 履歴書データはすべて **ブラウザの localStorage のみ** で完結する。
  サーバーは「Claudeへの中継」だけを担い、個人情報を保存・蓄積しない。

### 2.2 使いやすさ優先（ゼロフリクション）

- ログインなし＝アクセスした瞬間に使い始められることは、本サービスの
  **意図的な使いやすさ上の強み**。新規登録の離脱を生まない。
- 後述のセキュリティ対策（第8章）は、**この使いやすさを損なわないこと**を
  必須条件として設計する。セキュリティと使いやすさのトレードオフは
  「正規ユーザーには不可視な対策を優先する」方針で解決する。

## 4. MVPスコープ

### 4.1 含む（MVPで実装する）

1. チャットによる対話的な情報収集（履歴書/職務経歴書を選択 → 必要項目を順に質問）
2. Claudeによる構造化抽出（回答テキスト → 履歴書フィールドへマッピング）
3. ブラッシュアップ機能（志望動機・職歴・自己PRで原文 vs 修正案を提示 → 承認制）
4. 右パネルのリアルタイムプレビュー（履歴書 / 職務経歴書）
5. プレビュー上での直接編集（再編集機能）
6. 求人票・応募先テキストの貼り付け → 志望動機ブラッシュアップの文脈に利用
7. PDFダウンロード
8. 印刷（ブラウザ印刷 + 印刷用CSS）
9. localStorageへの自動保存・復元

### 4.2 含まない（次ステップに回す）

- 写真撮影/アップロード
- 音声入力
- AI採点スコア
- テンプレート/デザイン選択
- ログイン / DB保存（第2章の設計方針どおり、今回は実装しない）

| 領域 | 採用技術 |

## 6. アーキテクチャ概要

```
[左: ChatPanel]                    [右: ResumePreview]
  └ useChat (/api/chat)              └ resumeData state から描画
       │                                  ▲
       ▼                                  │
  Claude (tool use)  ──抽出──> resumeData ─┘
       │                          │
       └ブラッシュアップ提案 ──> 承認UI ──> resumeData反映
                                  │
                          localStorage 自動保存
```

- **会話設計**: Claudeにシステムプロンプトで「履歴書/職務経歴書の項目」を渡し、
  未入力項目を1問ずつ自然に質問させる。回答ごとにtoolを呼ばせて構造化データを返す。
- **ブラッシュアップ**: 自由記述項目（志望動機/職歴詳細/自己PR）が埋まったら、
  求人票テキストも文脈に含めて修正案を生成 → 差分UIで承認。
- **サーバーの役割**: Route Handler は Claude API への中継のみ。受け取った入力は
  リクエスト処理中だけメモリ上に存在し、応答後は破棄する（保存しない）。

## 8. セキュリティ要件（最重要）

APIキーを扱うため、セキュリティを重点的に設計する。
特に **ログインなし＝誰でもAPIエンドポイントを呼べる** ため、
「APIキー自体は隠せても、エンドポイント経由で課金は発生しうる」という
リスクに正面から対策する。

### 8.1 基本方針 — セキュリティと使いやすさの両立

- セキュリティ対策は原則 **サーバー側で完結し、正規ユーザーには不可視** とする。
- 体感フリクションを生みうるのは「レート制限」のみ。これは
  **通常利用では絶対に当たらない緩い閾値** に設定し、トレードオフを解消する。
- 入力上限・出力上限も、正規利用が途切れない **十分に余裕のある値** にする。

### 8.2 対策一覧（各対策の「使いやすさへの影響」を併記）

| 対策 | 内容 | 使いやすさへの配慮 |
| --- | --- | --- |
| **APIキーの秘匿** | `ANTHROPIC_API_KEY` はサーバー専用。`NEXT_PUBLIC_` を付けない／クライアントバンドルに含めない／Route Handler 内のみで参照。`.env.local` は `.gitignore` 対象、本番は Vercel 環境変数。ログ・エラーレスポンスに鍵を出力しない | 完全に不可視。影響ゼロ |
| **オリジンチェック** | `Origin` / `Referer` ヘッダを検証し、自サイト以外からの直接呼び出しを拒否 | 同一オリジンの正規利用には影響ゼロ |
| **レート制限** | IP単位・インメモリ方式（DB不要。MVPでは複数インスタンス間で厳密でなくてよい）。`lib/rate-limit.ts` に実装 | 履歴書1枚分の利用（チャット数十往復＋ブラッシュアップ数回）で **当たらない緩い閾値** に設定。万一到達時は「少し時間をおいて再開してください」と再開可能時刻を案内 |
| **入力上限** | リクエストボディサイズ・1メッセージの文字数・会話ターン数の上限 | 長い職務経歴や貼り付けた求人票が切れないよう、上限を **十分大きく** 設定 |
| **出力単価の固定** | `maxOutputTokens` と tool use のステップ数に上限を設定し、1リクエストあたりのコストを一定範囲に収める | ブラッシュアップ文や回答が途中で切れない **余裕ある値** に設定 |
| **Anthropic 側の上限** | Anthropic アカウントで月次の利用上限（spend limit）と使用量アラートを設定 | 不可視。影響ゼロ |
| **任意強化: BotID** | Vercel BotID によるボット遮断（DB不要） | 人間に CAPTCHA 等を出さないサイレント方式のみ採用。影響ゼロ |

### 8.3 エラー処理

- 失敗時、クライアントには内部エラーの詳細（スタックトレース・鍵・モデル名等）を
  返さず、汎用メッセージのみを返す。
- ただし文言は、ユーザーが次にどうすべきか分かる **親切なもの** にする
  （例:「通信に失敗しました。少し時間をおいて再試行してください」）。

### 8.4 入力データの取り扱い

- 履歴書に含まれる個人情報（氏名・住所・連絡先など）は **サーバーに保存しない**。
  リクエスト処理中のみメモリ上に存在し、応答後に破棄する。
- システムプロンプトとユーザー入力を明確に分離し、プロンプトインジェクション
  （ユーザー入力で指示を上書きされる攻撃）への耐性を持たせる。

## 9. 実装フェーズ

1. **基盤**: Next.jsプロジェクト作成、Tailwind/shadcn導入、2カラムレイアウト
2. **データモデル**: `resume-schema.ts` で履歴書/職務経歴書の項目を定義
3. **プレビュー**: resumeData → 履歴書/職務経歴書の描画（先にダミーデータで）
4. **チャット**: `/api/chat` + useChat、tool useで項目抽出 → resumeData更新
5. **セキュリティ**: APIキー秘匿・オリジンチェック・レート制限・入出力上限を実装（第8章）
6. **ブラッシュアップ**: `/api/brushup` + 承認ダイアログ、求人票入力欄を連携
7. **編集**: プレビュー項目のインライン編集
8. **保存**: localStorage自動保存・復元
9. **出力**: 印刷用CSS整備、印刷ボタン、PDFダウンロード
10. **仕上げ**: エラーハンドリング、空状態、Vercelデプロイ

## 10. 検証方法（End-to-End）

### 10.1 機能のゴールデンパス

`npm run dev` で起動し、ブラウザで以下を通す:

1. 履歴書/職務経歴書を選択 → チャットが質問を開始
2. 数項目に回答 → 右プレビューにリアルタイム反映される
3. 求人票を貼り付け → 志望動機のブラッシュアップ案が文脈を反映
4. ブラッシュアップ案を承認/却下 → プレビューに正しく反映
5. プレビュー項目を直接編集 → 値が更新される
6. リロード → localStorageから内容が復元される
7. 印刷プレビュー / PDFダウンロードでレイアウト崩れがないか確認

**エッジケース**: 未回答のまま出力、長文の職歴、APIエラー時の挙動

### 10.2 セキュリティ確認

- ビルド成果物（クライアントバンドル）に `ANTHROPIC_API_KEY` が **含まれていない**
  ことを確認（バンドル内文字列検索）。
- 別オリジンから `/api/chat`・`/api/brushup` を直接呼び出すと拒否されること。
- APIエラー時、レスポンスに鍵や内部詳細が漏れていないこと。

### 10.3 使いやすさ確認（セキュリティ対策がフリクションになっていないか）

- 履歴書1枚を最後まで作り切る通常利用で、**レート制限に一度も当たらない** こと。
- 長い職務経歴・長い求人票を貼り付けても **入力上限で切られない** こと。
- ブラッシュアップ結果が **出力上限で途切れない** こと。
- レート制限に達した場合でも、ユーザーに分かりやすい再開案内が出ること。
